How it works

A clean local stack, offline by design.

  • Local model runtime: Ollama or Foundry Local, serving an open model over a loopback endpoint only.
  • FastAPI backend: worklist, reporting, triage, RAG, and the agent, all in Python.
  • Local storage: SQLite plus local files, gitignored, holding studies, reports, and the knowledge base.
  • No-build web UI: a self-contained single-page app, no external assets, no internet required.

The model, picked and tested

Recommended local model: Qwen3-32B + MedGemma

Qwen3-32B is the primary reasoning model: Apache 2.0 licensed, with a thinking mode that suits impression drafting and structured reports. MedGemma 27B adds medical language and chest-X-ray-trained image understanding under Google's healthcare-friendly HAI-DEF terms.

  • Reasoning: Qwen3-32B (Apache 2.0, thinking mode) for reports, triage, and RAG.
  • Medical + images: MedGemma 27B multimodal, trained on chest X-rays and clinical text.
  • Runs on 32GB today: Qwen3-14B or gpt-oss:20b, paired with MedGemma 4B.

Tested on real hardware

Measured performance, not marketing.

Benchmarked on an Apple M1 Max, 32GB, using the same prompts the app uses. gpt-oss:20b ran 100% on the GPU at 33 to 41 generation tokens per second, using about 12GB. A Mac Studio adds memory and bandwidth for larger models and higher throughput.

Strong on report drafting and grounded answering. Triage scored lower because the model undercalled one case, which is exactly why RadHarness wraps triage in deterministic rules plus mandatory radiologist verification. Warm-run figures, GPU (Metal).
